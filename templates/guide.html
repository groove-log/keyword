Overview

시스템 개요

IRDA(Intelligence Risk Detection & Analysis)는 편의점/유통 기업 고객센터에 접수되는 한국어 상담 텍스트에서 위험 신호를 자동으로 탐지하고 우선순위를 결정하는 AI 분류 엔진입니다. BGE-M3 다국어 임베딩 모델(llama.cpp HTTP 서버 via http://localhost:8081)을 기반으로 하며, 단순 키워드 매칭이 아닌 의미 유사도 기반으로 동작합니다.

핵심 입력

고객 상담 텍스트 (한국어 자유 형식)
단일 문장 ~ 수 문단 처리 가능

핵심 출력

위험 카테고리 + 외부압박 카테고리 + 시급성 3축 판정
위험 키워드 목록 + 심각도 1~10점 + tone/flags

주요 기술 스택

BGE-M3 (BAAI/bge-m3, llama.cpp) · Kiwi 형태소 · FastAPI · Celery + Redis · PostgreSQL 15

Core Engine

SemanticClassifier 상세

classify_danger(text, custom_thresholds)

X축 위험 카테고리를 판정합니다. BGE-M3로 텍스트를 임베딩하여 5개 카테고리 지침 임베딩과 코사인 유사도를 계산하고, 베이스라인 보정 후 임계값을 적용합니다.

임계값 결정 규칙
  • 위해 렉시콘 직접 매칭 시: DANGER_KEYWORD_LINKED_THRESHOLD (기본 0.10) 적용 → 민감도 최대화
  • 렉시콘 매칭 없음 시: DANGER_THRESHOLDS[카테고리] 또는 DANGER_THRESHOLD (기본 0.20) 적용
  • 반환: 통과 카테고리를 점수 내림차순 정렬한 리스트. 통과 없으면 [("정상 문의", best_score)]

classify_external(text, custom_thresholds)

Y축 외부압박 카테고리를 판정합니다. "법적/민원제기"와 "언론제보" 2개 카테고리에 대해 classify_danger와 동일한 임계값 로직으로 동작합니다.

classify_urgency(text)

Z축 시급성을 판정합니다. Kiwi 형태소 분석 후 어휘 집합 매칭과 과거형 감지를 순서대로 수행합니다. 반환값: "IMMEDIATE" | "SHORT-TERM" | "MONITOR"

classify_phrase(phrase, custom_thresholds, doc_risk_level, doc_external_level, custom_linked_threshold, custom_min_safeguard)

개별 키워드/구문의 카테고리를 판정합니다. 위험 + 외부 전체 7개 카테고리에 대해 임베딩 유사도를 계산하고, 문서 수준 판정 결과와 렉시콘 매칭 여부를 결합하여 임계값을 동적으로 결정합니다.

Safeguard 임계값 로직
  • 렉시콘 직접 매칭: linked_threshold 적용
  • 문서 위험 판정과 카테고리 일치: min(linked_threshold, min_safeguard_score) 적용
  • 기타: 카테고리별 개별 임계값 적용
  • 보호단어에 해당하면 즉시 "정상", 0.0 반환 (모든 임베딩 계산 스킵)

NLP Core

BGE-M3 베이스라인 보정

BGE-M3는 한국어 텍스트 간 코사인 유사도가 서로 무관한 문장 사이에도 0.65~0.73 범위로 높게 나타납니다. 이 "플로어(floor)" 현상을 보정하지 않으면 정상 문의도 위험으로 오분류됩니다.

핵심 문제

"영업시간 문의합니다"와 "폭행 카테고리 지침 설명" 사이의 코사인 유사도도 약 0.68 수준. 임계값을 0.7 이상으로 설정하면 실제 위험도 탐지 못하고, 낮게 설정하면 모든 정상 문의가 탐지되는 딜레마.

해결: 카테고리별 베이스라인 차감 정규화

12개 중립 텍스트와 각 카테고리 지침 임베딩의 코사인 유사도 평균을 해당 카테고리의 baseline으로 사용합니다.

adjusted = max(0.0, (raw_score - baseline) / (1.0 - baseline))

이 수식은 유효 신호 구간 [baseline, 1.0]을 [0.0, 1.0]으로 재정규화합니다. 임계값 0.20은 "중립 대비 20% 이상 높은 의미 유사도"를 의미하게 됩니다.

중립 텍스트 12개 (베이스라인 측정용)

"POS 시스템 접속이 되지 않습니다. 빠른 조치 원합니다."
"입고된 바나나 상태가 좋지 않습니다. 반품 요청드립니다."
"영업시간 변경 안내 부탁드립니다"
"포인트 적립이 안 되는데 확인 부탁드립니다"
"영수증 재발급 요청드립니다"
"제품 사용 방법에 대해 질문하고 있습니다"
"강남점 주차장 이용 방법이 궁금합니다. 빠른 안내 부탁드립니다."
"배송이 지연되고 있습니다. 빠른 처리 요청드립니다."
"개선되지 않으면 불편함을 느낍니다. 빠른 조치 부탁드립니다."
"매장 청결 상태 개선을 요청합니다. 신경 써주세요."
"제품 불량으로 교환을 요청합니다. 빠른 처리 바랍니다."
"고객 응대가 불친절했습니다. 개선 요청드립니다."
지연 로딩 (_ensure_cached)

베이스라인 계산은 최초 분류 요청 시 1회만 수행됩니다 (7개 카테고리 × 12개 중립 텍스트 = 84회 코사인 유사도 계산). 결과는 cached_baselines 딕셔너리에 캐시됩니다. llama.cpp 서버가 반드시 준비되어 있어야 하며, 서버 재시작 시 캐시가 초기화됩니다.

Keyword Extraction

키워드 추출 엔진

위험 판정 후, 어떤 단어가 위험 신호를 유발했는지 CS 상담원에게 시각적으로 보여주기 위해 위험 키워드를 추출합니다. extract_keywords() 함수가 담당합니다.

후보 추출 전략 (5가지)

1
단일 명사 (NNG, NNP)
Kiwi 분석에서 2글자 이상 일반명사·고유명사 추출
2
인접 명사 N-gram
연속된 명사를 합성하거나 띄어쓰기 변이를 생성하여 복합 명사 후보 추가
3
동사/형용사 명사화
VV/VA 어간에 종성 계산 후 "ㅁ/음" 결합 (예: 다치다 → 다침, 쓰러지다 → 쓰러짐)
4
관형어 + 명사구
VV+ETM(관형형 어미) + 후속 명사 조합 (예: 쓰러진 손님, 미끄러진 바닥)
5
명사구 + 동사 결합
인접 명사구와 바로 뒤 동사 어간을 결합 (예: 응급실 이송, 구급차 호출)

점수 계산 및 가이드 가중치

# 기본 점수: 후보와 문서 전체의 코사인 유사도
base_score = cosine_similarity(candidate_emb, doc_emb)

# 위험 문서일 때만 가이드 버프 활성화
# (doc_risk_level != "정상 문의" OR doc_external_level != "정상 문의")
guide_score = max(cosine_similarity(candidate_emb, guide_emb) for all 7 guides)

# 렉시콘 직접 매칭 시 1.5배 가속
if 렉시콘 매칭: final = base + (weight_coeff * guide_score * 1.5)
else:           final = base + (weight_coeff * guide_score)

# weight_coeff 기본값: 0.35 (env: DANGER_KEYWORD_GUIDE_WEIGHT)
# 보호단어는 가이드 버프 완전 무효화

배치 처리 및 중복 제거

  • 배치 크기 32: 후보를 32개씩 청크 분할 → llama.cpp 배치 임베딩 요청 (VRAM 초과 방지)
  • 폴백: 배치 실패 시 후보를 1개씩 개별 요청 (Plan B)
  • 중복 제거: 띄어쓰기 정규화(spaceless key) → 부분 문자열 포함 체크 → 더 구체적인 복합어가 단순 명사를 대체
  • 최종 반환: 점수 내림차순 top_n개 (기본값 top_n은 요청 파라미터)
Severity Engine

심각도 점수 엔진 (classify_severity)

같은 "안전사고" 카테고리라도 "어제 계단에서 미끄러질 뻔했어요"와 "지금 고객이 쓰러졌고 구급차 불렀습니다!!!"는 완전히 다른 대응 수준이 필요합니다. 심각도 엔진은 1~10점 척도로 이를 정량화합니다.

반환값: {"score": int, "tone": str, "escalation": float, "flags": list[str]}

점수 계산 방식 (순서대로 적용)

# Step 1: 극단 Critical 키워드 → 즉시 10점 + flag="critical_keyword"
critical_words = {사망, 자살, 자해, 죽다, 살해, 119, 구급차, 의식불명, 혼절, 실신, 중태, 위독}

# Step 2: 카테고리별 기본점수 (가장 높은 1개 채택)
{폭력/폭행: 6, 성폭력: 7, 이물질 상품: 5, 식품위생: 5, 안전사고: 6, 법적/민원제기: 4, 언론제보: 5}

# Step 3: 시급성 가중치
IMMEDIATE → +3점,  SHORT-TERM → +1점,  MONITOR → 0

# Step 4: 어조·위협 보너스
공격적 어조(aggressive_words 집합 매칭)    → +1  | tone="aggressive"
법적 위협(법원, 변호사, 경찰, 신고 등)     → +2  | flag="legal_threat"
언론 위협(유튜브, 방송, SNS, 커뮤니티 등)  → +2  | flag="media_threat"
느낌표 3개 이상(!!!)                       → +1  | flag="high_emotion"

# Step 5: 다중 카테고리 보너스
카테고리 3개 이상 → +2 | flag="multi_category"
카테고리 2개      → +1 | flag="multi_category"

# 최종: min(10, 합산)

심각도별 대응 기준

10
Critical — 사망·자살·자해 등 critical 키워드 감지 (즉시 10점)
즉각 CSO + 법무팀 에스컬레이션
8~9
Extreme — 다중 위협 + IMMEDIATE + 공격적 어조 조합
PR + Legal + QA 긴급 배정 (SLA 1h)
5~7
High — 위험 카테고리 + 법적/언론 위협 조합
CS 총괄 매니저 배정 (SLA 24h)
1~4
Moderate — 단순 위험 언급 또는 일반 불만
CS 선임 상담원 배정 (SLA 48h)

Feedback Loop

피드백 루프 & 임계값 자동보정

CS 상담원이 AI 판정이 틀렸다고 직접 신고할 수 있습니다. 누적된 피드백 데이터를 분석하여 카테고리별 임계값 조정 방향을 자동으로 제안합니다.

피드백 수집

대시보드 각 행의 ✓ / ✗ 버튼으로 피드백을 제출합니다. 오분류 신고 시 실제 카테고리와 메모를 입력할 수 있습니다.

# POST /api/feedback/{counseling_id}
{
  "is_correct": false,
  "actual_danger": "식품위생",      # 상담원이 판단한 실제 위험 카테고리
  "actual_external": "정상 문의",
  "actual_urgency": "MONITOR",
  "note": "단순 부패 불만, 법적 위협 없음"
}

임계값 보정 제안 로직

GET /api/threshold-calibrate는 counseling_feedback를 counseling_data에 JOIN하여 카테고리별 정확도를 계산합니다 (최소 5건 이상 피드백 필요).

조건제안 방향조정 폭의미
정확도 < 60%임계값 상향+0.03오탐 과다 → 더 엄격하게
정확도 > 90%임계값 하향-0.02여유 있음 → 더 민감하게
60% ~ 90%유지0현 임계값 적절
주의: 자동 적용되지 않음

보정 제안은 조언입니다. 제안을 검토 후 .env의 DANGER_THRESHOLDS를 수동 수정하고 재배포해야 적용됩니다. UI 하단 "🔬 임계값 보정 제안" 버튼으로 현재 제안 내용을 확인합니다.

Pipeline

데이터 파이프라인

동기 분석 경로 (⚡ 실시간)

POST /api/extract/sync로 요청 → FastAPI 메인 스레드에서 즉시 처리 → 결과 반환. DB 저장 없음. UI 슬라이더 테스트에 사용됩니다.

비동기 분석 경로 (📥 Celery 큐)

1
POST /api/extract/async 접수
counseling_data에 status='PENDING' 행 INSERT → counseling_id 즉시 반환
2
Celery 워커 처리
Redis 큐에서 태스크 수신 → 3D 분류(X·Y·Z) + classify_severity + extract_keywords 순서 실행
3
DB 업데이트
keywords, risk_level, external_issue, urgency_level, detected_categories(severity 포함), status='COMPLETED' 업데이트
4
자동 재시도 (max 3회)
ConnectionError / TimeoutError / OSError 발생 시 5초 대기 후 재시도. 3회 모두 실패 → status='FAILED' 기록
5
UI 폴링 (3초 간격)
대시보드가 3초마다 /api/history 폴링 → PENDING → COMPLETED 상태 변화 실시간 반영

상태값 (status)

값의미UI 표시
PENDING큐 대기 중 또는 처리 중🟡 노란 점
COMPLETED분석 완료🟢 초록 점
FAILED3회 재시도 후 최종 실패🔴 빨간 점

Tuning Guide

임계값 튜닝 가이드

모든 임계값은 .env의 DANGER_THRESHOLDS에서 관리합니다. 변경 후 반드시 재배포해야 적용됩니다. UI 슬라이더는 재배포 없이 임시 테스트에 사용합니다.

임계값 의미 (베이스라인 보정 후 기준)

값을 낮출 때 (예: 0.27 → 0.22)
  • 더 많은 케이스를 위험으로 탐지 (민감도 ↑)
  • 오탐(정상 → 위험 오분류) 증가 위험
  • 피드백 정확도 > 90%일 때 고려
값을 높일 때 (예: 0.22 → 0.27)
  • 탐지 건수 감소 (특이도 ↑)
  • 미탐(위험 → 정상 오분류) 증가 위험
  • 피드백 정확도 < 60%일 때 고려

슬라이더 범위 (UI 기준)

슬라이더범위현재 .env 기본값의미
카테고리 임계값0.30 ~ 0.95 (step 0.01)카테고리별 0.22~0.28각 카테고리 탐지 민감도
연관 키워드 임계값 (linked)0.30 ~ 0.80 (step 0.01)0.10렉시콘 매칭 시 완화 임계값
안전망 임계값 (safeguard)0.30 ~ 0.80 (step 0.01)0.15문서 판정 일치 카테고리 최소 임계값
